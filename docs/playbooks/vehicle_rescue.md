# playbook · 车辆应急救援

> 适用触发词：救援、拖车、清障、被困、事故车辆、道路救援、施救力量。
> 场景对齐：比赛场景 3（车辆应急救援）。
> 推荐报告模板：`reports/decision_memo.md`（决策建议书 / 调派建议）。

---

## 一、预报（forecast）

**目标**：在事件发生前后，快速给出可用救援力量清单 + 最近点位。

数据源与调用：

- `await window.weatherPlatform.resources({ province })`：一次拿到 `vehicleRescuePoints`（车辆救援点）+ `materialReserves`（物资）+ `riskSegments` + `hazardPoints`。
- `await window.weatherPlatform.summary({ province })`：拿气象态势，判断是否有次生风险。
- 事件位置：由用户提供（路线代码 + 桩号）；若用户只说了大致位置，先请求补齐。

关键字段：

- `vehicleRescuePoints`：`name / province / city / district / lon / lat / vehicles / phone`。
- `materialReserves`：`name / lon / lat / stockpile`（如平台字段名不同，adapter 会保留原样）。

---

## 二、研判（analyze）

分步：

1. **事件锚点**：明确 `<路线> <桩号> <省份>`；如为经纬度，反算属地。
2. **候选救援点**：
   - 从 `vehicleRescuePoints` 里筛所在省 + 相邻省的记录。
   - **距离粗算**：用经纬度 haversine 或 `等经纬度差 × 111 km` 粗算；db 不跑真实路网规划，明确标注"仅经纬度直线距离，实际到达时间需接入路径规划服务"。
   - 排序，取最近 3–5 个候选点。
3. **物资匹配**：
   - 事件类型 → 需要物资类别：翻车 / 侧滑 → 拖车 + 破碎机；洪水冲毁 → 沙袋 + 排水泵；结冰 → 防滑链 + 融雪剂。
   - 从 `materialReserves` 找对应类别，取最近 1–2 处。
4. **气象叠加**：
   - 结合 `summary` 判断救援路线沿线是否处于红/橙预警。
   - 如有沿线预警：救援车辆要注意二次风险，提示"救援车队沿线是否也需要防护"。

---

## 三、决策（decide）

给出建议方案，允许人工调整：

- **推荐调派**：`<救援点名称>`（距离 ~X km，直线）+ `<车辆数 / 联系电话>`。
- **备选调派**：2 号备份 + 3 号备份。
- **物资伴随**：`<物资库>` 的 `<物资类别>`。
- **风险提示**：若沿线仍有降雨/大雾/结冰，写清"救援车辆需 + <防护建议>"。

明确边界：

- db **不下达调派指令**，也不代替调度员挑最终方案。
- db **不出具真实路网导航时间**；只给经纬度直线距离作为参考。
- 联系电话仅从平台数据回显，禁止编造。

---

## 四、处置（act）

db 侧动作：

1. `renderStatus(bundle)`：把候选救援点/物资库列在 Canvas 卡片。
2. 出决策建议书 `reports/decision_memo.md`：条目化列出推荐 / 备选 / 风险提示。
3. 若用户希望地图核对：`openBigScreen('threeMap')` 打开 3D 大屏，值班员可自行叠加图层查看救援点分布。
4. 附上原始数据源（`resources` 接口 + 数据模式：live/demo）。

---

## 五、评估（review）

事件闭环后：

- 记录实际到达时长与直线距离偏差，供后续经验值参考。
- 记录本次救援消耗物资，反哺"物资类别匹配"经验。
- 记录未命中问题（例如平台里救援点电话不通），标记待清洗。
- 用 `reports/after_action.md` 出复盘小节，重点写"距离粗算 vs 实际路网"的差异。

---

## 附：常见坑与提醒

- **直线距离 ≠ 实际到达时间**：db 输出中一定要标"直线粗算"字样，避免评委或值班员误读。
- **多省交界**：事件靠近省界时，把相邻省的救援点也纳入候选。
- **数据缺失**：如 `vehicleRescuePoints` 空，明说"当前该省救援点数据未推送/未开放"，不要编造。
- **隐私**：救援点电话属于内部信息，可以在决策建议书里列出，但**不要**写进公开通报模板。
