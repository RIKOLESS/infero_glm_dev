# playbook · 突发事件应急处置

> 适用触发词：应急、突发事件、启动响应、分级响应、事件处置、灾情核实。
> 场景对齐：比赛场景 2（突发事件应急处置）。
> 推荐报告模板：`reports/incident_report.md`（事件通报）或 `reports/decision_memo.md`（决策建议书）。

---

## 一、预报（forecast）

**目标**：判断当前是否具备"启动应急响应"的触发条件，避免为空报警启动。

数据源与调用：

- `await window.weatherPlatform.summary({ province })`：拿到预警级别 + 风险路段总量。
- `await window.weatherPlatform.segmentRisk({ province })`：风险路段、隐患点、连续降雨风险区。
- `await window.weatherPlatform.resources({ province })`：救援点 + 物资储备（供决策阶段用）。
- `await window.weatherPlatform.earthquake()`：如果事件类型是地震，取最新震例（详见 `earthquake_survey.md`）。

判定触发条件（满足任一即建议进入应急流程）：

1. 出现 **红色气象预警** 且落在国高/省高主干路网。
2. 出现 **中央气象台三类风险预警**（河洪 / 地质 / 山洪，`riskWarningLatest`）叠加路网。
3. 平台大屏出现 **多灾种耦合**（例如降雨 + 大风、降雨 + 大雾）。
4. 用户主动告知发生阻断、事故、灾情（此时以人工输入为准）。

---

## 二、研判（analyze）

分四步：

1. **事件性质分类**：
   - 单灾种（暴雨/大风/大雾/大雪/冰冻）
   - 多灾种耦合
   - 次生灾害（滑坡/泥石流/塌方）
   - 交通类（重大事故/大范围拥堵）
2. **影响范围**：
   - 省/市/区县、涉及路线代码、桩号区间；引用 `affectedTop10 / keySegments`。
   - 涉及人口 / 车流：如无数据，就说"暂无平台侧车流数据，须与路政交警联动核实"。
3. **响应级别建议**：
   - 参照 4 级响应体系（I 特别重大 / II 重大 / III 较大 / IV 一般）。
   - 4 级判定按"影响路网层级 + 预警等级 + 是否叠加隐患点"综合给出，避免僵硬照搬。
4. **数据质量说明**：
   - 明确列出接口成功/空/失败清单；对于失败接口需要人工在平台复核的，写进"待人工确认项"。

---

## 三、决策（decide）

给三个层次的建议，让值班主任按需勾选：

### A. 立即执行
- 启动 **X 级响应**（给出等级 + 触发依据）。
- 指定 **牵头处室**（例如应急指挥中心 / 路网调度中心 / 气象服务处）。
- 通报路径：内部群 → 沿线路政 → 交警 → 属地政府 → 上级主管部门。

### B. 短期部署（0–6h）
- 前置救援力量：由 `resources` 中的 `vehicleRescuePoints` 结合影响路段，粗算最近的 2–3 个救援点。
- 前置物资：`materialReserves` 中最近的储备库；建议物资类型（沙袋、防滑链、破碎机…）。
- 大屏展示：`openBigScreen('shortTerm')` 或 `openBigScreen('threeMap')`。

### C. 中期跟踪（6–24h）
- 每 1h 复查一次短临排行；预警降级前不解除响应。
- 每 3h 出一次值班简报（走 `reports/duty_brief.md`）。
- 事件全过程留痕，便于事后复盘。

---

## 四、处置（act）

db 侧动作：

1. `renderStatus(bundle)`：把研判过程与数据源挂到 Canvas，便于值班员回看。
2. 输出 **事件通报（`reports/incident_report.md`）**：标注事件性质 / 响应级别 / 影响面 / 联动对象 / 时间线。
3. 若用户要求，用 `openBigScreen('analysisReport')` 打开分析看板给现场调度用。
4. 明确告知 db 的边界：**路径规划、力量调度指令下达不由 db 完成**，只提供建议；正式指令须走平台指挥台或线下签批。

处置期沟通模板（可插入通报）：

- "根据 `<数据来源>`，识别 `<影响路段清单>` 处于 `<最高预警级别>`。已建议启动 `<X 级响应>`，牵头 `<单位>`，通报 `<对象>`。请值班主任签批。"

---

## 五、评估（review）

响应结束后：

- **过程留痕**：从 db 侧 Canvas 卡片、简报、报告导出（`exportReport` 生成 HTML）。
- **成效指标**（选做，视情报支持情况）：
  - 预警提前量（分钟）
  - 阻断时长（分钟）
  - 救援到达时长
  - 处置期人员/物资投入
- **改进项**：数据接口空/失败原因、内部通报断点、大屏使用体验。
- 输出 **复盘评估**（`reports/after_action.md`），归档到值班日志与月度总结。

---

## 附：db 的价值边界

- ✅ **能做**：数据汇总、分级判定建议、通报文本生成、平台驾驶舱操作、简报导出。
- ⚠️ **需人工确认**：响应级别正式启动、跨部门协调、指令下达、灾情核实。
- ❌ **不做**：不伪造现场数据、不代替指挥中心决策、不下达强制指令。
